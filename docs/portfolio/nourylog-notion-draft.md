## 프로젝트 개요

nouryLog는 식단 기록 앱을 소재로, AI 보조 개발 워크플로우(PRD → 명세 → Human Gate → 검증)를 실제 제품에 끝까지 적용한 사례입니다.

서버·모바일 앱·관리자 웹으로 이루어진 3-tier 제품을 OpenAPI 계약과 공유 API 클라이언트로 연결하고, 요구사항 문서·계약·승인·테스트를 완료 조건으로 두어 AI가 생성한 코드를 검증·책임지는 흐름을 만들었습니다. 사용자는 모바일 앱에서 식단을 기록하고 영양표 OCR로 영양성분을 보조 입력하며, 관리자는 관리자 웹에서 운영 데이터를 관리합니다.

현재 Google Play 비공개 테스트를 진행하며, 로컬 개발을 넘어 EAS Build·Railway 배포·실기기 API 연동까지 검증하고 있습니다.

흐름 요약: 모바일/관리자 웹 → Express API → PostgreSQL · OCR 외부 API · SNS 로그인 검증 → 통일된 API 응답 형식에 맞춰 화면 반영.

## 주요 기능

#### 식단 기록

모바일에서 식사 추가·조회·수정·비활성화. 음식 템플릿 기반 영양 계산, 끼니·섭취량 입력 모드 지원.

#### 영양표 OCR

사진 촬영·앨범 선택 후 OCR API로 칼로리·탄수화물·단백질·지방 및 신뢰도를 받습니다. 신뢰도가 낮으면 수동 확인을 유도하고, 실패 시 수동 입력으로 전환합니다.

#### 식단 인사이트

누적 기록 기반 규칙 인사이트. 홈 요약 카드, 주간 리포트(최근 7일), 월간 패턴 카드. LLM 호출 없이 규칙 템플릿으로 생성하고 면책 문구 표시.

#### 사용자 모바일 앱

Expo(React Native). 홈·기록·통계·설정과 음식 검색·섭취 이력, 체중 추세, 알림(리마인더), 다크모드 제공. 네이버/카카오/구글 네이티브 SDK 로그인 후 서버에서 서비스 토큰 발급.

#### 관리자 웹

Vite + React. 대시보드(통계·재집계), 회원·음식·문의·공지 목록 — 필터·검색·초기화·15행 페이지네이션.

#### 서버 API

Express + Prisma(PostgreSQL). 인증·식단·OCR·통계·인사이트·관리자 API. 오류 응답 필드와 HTTP 상태를 계약에 맞춰 통일했습니다.

#### 배포·운영

서버 Railway, 모바일 EAS Build, Android 패키지 com.nourylog.app 기준 Play 비공개 테스트.

## Problem

초기에는 기획을 문서 없이 Cursor AI와 대화하며 바로 구현했습니다. 빠르게 결과물을 만들 수 있었지만, 서버·모바일 앱·관리자 웹이 함께 연결되면서 다음 문제가 반복되었습니다.

- 요구사항이 고정되지 않아 작업 중 구현 방향이 흔들림
- AI가 초기 의도와 다른 기능·구조를 제안하거나 범위를 임의 확장
- 존재하지 않는 API·파일을 전제로 구현이 진행됨
- 서버 응답 구조와 모바일·웹의 사용 방식이 어긋남
- 디자인·기능을 무검토로 확정하는 경우
- PRD·결정 사항·작업 기록이 여러 위치에 분산

단순히 “AI와 대화하며 구현”만으로는 3-tier 제품의 일관성·검증·재현을 안정적으로 관리하기 어렵다고 판단했습니다.

---

## Analysis

#### 1. 제품은 서버·모바일·관리자 웹 분리 + 계약 연동이 필요

클라이언트마다 API를 따로 해석하면 응답이 조금만 바뀌어도 전체가 깨집니다. 모노리스 UI보다 3-tier 분리와 OpenAPI·공유 클라이언트로 타입·호출 방식을 맞추는 편이 낫다고 봤습니다.

#### 2. 구현 전 PRD·API 계약을 SSOT로 고정

PRD, API 계약, 상태/오류 UX 매핑을 요구사항 문서로 두고, PRD 승인 전 구현·목업 착수를 하지 않는 Gate를 적용했습니다. 기능 확장도 같은 계약 흐름(버전별 API 델타 문서)으로 누적했습니다.

#### 3. AI 작업 경계는 AGENTS.md·역할 분담

AGENTS.md와 frontend/backend/prd/qa 역할 정의로 AI가 임의로 구조를 바꾸거나 승인 없이 UI 병렬 구현에 들어가지 않도록 했습니다. UI 신규는 디자인 승인 후, FE/BE 병렬은 Gate 2(계약·디자인 확정) 후만 허용합니다.

#### 4. 승인 지점(Human Gate)은 단일이 아님

PRD 승인, 디자인 A/B 선택, Gate 2, rules/skills 수동 반영 등 여러 체크포인트에서 사람이 확인한 뒤 다음 단계로 넘깁니다. 디자인 승인 = 구현 착수 승인으로 통합했습니다.

#### 5. AI 기능도 검증·재현 기준으로 관리

생성형 기능이라도 응답의 일관성·재현·면책을 관리하지 못하면 제품으로 책임지기 어렵다고 판단했습니다. 그래서 AI 기능도 PRD·계약·테스트로 동작 기준을 고정한 뒤에만 확정했습니다.

#### 6. 품질은 테스트·스모크·연동 체크리스트로 확인

서버 단위 테스트, 스모크 체크리스트, 엔드포인트별 연동 매핑 표로 “돌아간다”를 완료 조건으로 두었습니다.

#### 7. 모바일·출시는 웹 튜토리얼과 다른 검증이 필요

Expo Go vs EAS Development Build, Play 앱 서명 SHA-1, 네이티브 SNS SDK 등은 문서·회귀 시나리오 없이는 재현이 어렵습니다. SNS·로그인은 네이티브 SDK와 토큰 교환 API를 기준으로 잡고, 앱 버전 업데이트·출시 전 회귀 시나리오로 고정하는 쪽이 맞다고 판단했습니다.

## Action

#### 1. 3-tier 프로젝트 구조 및 OpenAPI 계약

Express + Prisma 서버, Expo 모바일, Vite 관리자 웹, 공유 API 클라이언트를 구성했습니다. OpenAPI 스냅샷과 요구사항 문서를 계약의 단일 기준으로 유지합니다.

#### 2. 영양표 OCR 및 경계값 테스트

OCR API 연동, 무료 사용량·기간 정책, 피드백 경로에 단위 테스트를 두었습니다.

#### 3. SNS 로그인 — 웹 OAuth에서 네이티브 SDK로 전환

초기에는 Chrome Custom Tab + 서버 OAuth 리다이렉트 패턴이 검토되었습니다. 모바일·배포 전제를 대조한 결과 Expo Go·EAS Build·콘솔 SHA-1 제약과 맞지 않아 네이티브 SDK + 토큰 교환 API로 전환했습니다. 3사×4 시나리오 회귀 기준을 문서로 고정했습니다.

#### 4. 식단 인사이트 — 검증 가능한 형태로 방향 고정

식단 인사이트는 여러 구현안을 검토한 뒤, 개인 데이터에 대한 응답 일관성·면책·재현을 우선해 LLM 호출 없는 규칙 기반 주간·월간 리포트로 방향을 고정했습니다. 같은 입력에 같은 결과가 나오고 근거를 함께 보여주도록, 주간 집계는 KST 롤링 7일을 기준으로 계약화했습니다.

#### 5. 배포 파이프라인 및 Play 비공개 테스트

Railway(서버), EAS Build(모바일), Play Console 비공개 테스트 트랙. 출시 0단계·SNS 콘솔 등록·환경 변수 체크리스트와 앱 버전 업데이트(권장/강제) 흐름을 정리했습니다.

#### 6. PRD·문서 세트 및 Gate 운영

기준 PRD에서 API 계약, 상태 매핑, QA 시나리오, Stage 3 체크리스트로 문서 세트를 분리했습니다. Gate 2 충족을 기록한 뒤 병렬 구현에 착수했습니다.

#### 7. 디자인 A/B 비교 및 Human Gate

Stitch 시안(안 B)과 자체 목업(안 A)을 비교표로 정리하고 안 B(Stitch)를 선택한 뒤 구현했습니다. 무검토 UI 확정을 막기 위한 승인 단계입니다.

#### 8. API 연동 1:1 점검

엔드포인트별로 서버 구현·모바일·관리자 웹 호출 위치를 매핑해 응답 구조와 사용 흐름이 어긋나지 않게 확인했습니다.

#### 9. 작업 기록 자동화 및 AI 운영 Kit

요구사항 문서 변경 시 Obsidian 볼트 동기화 Hook, 위험 명령 차단 Hook, 역할별 Subagent, 검증·변경 문서화 Skill, agents/rules/skills 템플릿 저장소로 다른 PC·프로젝트에서도 동일 AI 작업 환경을 재현할 수 있게 했습니다. rules 개선은 후보로만 기록하고 사람 승인 후 반영합니다.

## Result

- 서버·모바일·관리자 웹을 OpenAPI 계약으로 연결한 제품형 3-tier 앱을 구현·연동했습니다.
- PRD·Gate·다중 Human Gate·연동 체크리스트·테스트로 AI 보조 구현 시 요구사항 이탈·구조 불일치를 줄였습니다.
- AI 기능도 일관성·재현이 검증되는 형태(규칙 기반 인사이트)로 동작 기준을 고정했습니다.
- Google Play 비공개 테스트로 실제 빌드·기기·API·환경 설정 흐름을 검증 중입니다.
- OpenAPI·문서 세트·Hook 기반으로 운영·인수인계·재현 가능한 구조를 갖췄습니다.

## 이 프로젝트에서 강조하고 싶은 점

이 프로젝트는 “Cursor로 앱을 만들었다”는 수준이 아니라, AI가 생성한 코드를 3-tier 제품 안에서 어떻게 검증·책임지는지를 보여주기 위한 경험입니다. Cursor AI로 구현하되 PRD·계약·승인·테스트를 완료 조건으로 두었습니다.

rag-assistant에서 RAG_EVAL로 검색·응답 정책을 수치화했듯, nouryLog에서는 PRD·OpenAPI 계약·Human Gate·연동 체크리스트·테스트를 완료 조건으로 두었습니다. 서버 응답이 바뀌면 모바일·웹이 함께 깨지는 구조이므로, “구현”과 “계약 일치 확인”을 분리하지 않았습니다.

SNS 로그인 전환 사례처럼, AI가 흔히 제안하는 웹 OAuth 패턴이 모바일·출시 제약과 맞지 않을 때 계약·PRD·회귀 문서로 방향을 고정한 경험이 있습니다.

코드 공개 범위는 프로젝트마다 다르지만, AGENTS.md·PRD 목차·Obsidian 작업 기록·Play 테스트 화면 등으로 “문서화·검증 체계가 실제로 돌아갔다”는 증거를 포트폴리오에 첨부할 수 있습니다.

Java/Spring 역량용이 아니라, AI 보조 환경에서 제품을 끝까지(로컬 → 배포 → 스토어 테스트) 가져가는 흐름을 정리한 개인 프로젝트입니다.
