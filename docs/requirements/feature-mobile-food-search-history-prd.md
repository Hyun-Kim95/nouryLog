# 모바일 음식 검색·섭취 이력 PRD v0.1 (approved)

## 0) 전제·가정

| 항목 | 값 |
|---|---|
| 사업자 | 없음 (개인사업자·법인·사업자등록 미보유) |
| 수익 | 광고 + 후원 수준만. 본 기능은 수익 모델과 무관 |
| 결제·정산 | 본 PRD 범위에 없음 |

본 문서는 **PRD 초안 작성 단계까지**를 범위로 한다. 디자인(이중안) 승인 전 구현·병렬 구현에 착수하지 않는다.

## 1) 목적

- 사용자가 **음식명을 검색**해서 그 음식을 **언제 먹었고(이력)**, **선택한 기간 안에 몇 번 먹었는지(빈도)**를 한 화면에서 확인하게 한다.
- "이 음식 마지막으로 언제 먹었지?", "이번 달에 라면 몇 번 먹었지?" 같은 **자기 행동 인식(self-awareness)** 을 지원해 식단 관리 동기를 강화한다.
- 기존 식사 데이터(`Meal`)만으로 구현하며, 새로운 데이터 적재나 마이그레이션을 요구하지 않는다.

## 2) 적용 범위

- 대상: `apps/mobile`, `apps/server`.
- 포함:
  1. 음식 검색 입력(자동완성 포함).
  2. 검색어로 매칭된 식사를 **시간 역순 이력 리스트**로 표시(날짜 + 끼니 구분 포함).
  3. 선택한 **기간 내 섭취 횟수(빈도)** 요약 표시.
  4. 서버: 기존 `GET /me/meals`에 이름 검색 파라미터(`q`) 추가.
  5. 자동완성: 기존 `GET /me/meal-entry-suggestions` 재활용.
- 제외(본 PRD 비범위):
  - 음식명 **정규화·그룹핑**(예: "라면" 검색 시 신라면/짜파게티 분리 집계). 본 PRD는 **검색어 한 덩어리(`contains`) 매칭**으로 한정.
  - 기간별 추세 그래프(점점 자주/뜸하게).
  - 영양/칼로리 누적 통계(별도 InsightScreen 책임).
  - 검색 결과에서의 식사 수정·삭제(이력은 읽기 전용; 편집은 기존 화면 흐름 활용 — D-5 참조).

## 3) 사용자 흐름

### 3.1 진입
1. 통계(Stats) 탭 상단의 "음식 검색" 입력/버튼을 탭 → 별도 검색 결과 화면(스택 라우트 `FoodSearchScreen`)으로 push.
2. 검색어가 없을 때는 안내 문구 + "최근 먹은 음식" 칩을 노출(칩 탭 시 그 이름으로 바로 검색).
3. 검색 입력에 음식명 일부 입력 → 자동완성 후보 노출.
4. 후보 선택 또는 검색 실행.

### 3.2 결과 화면 구조 (와이어 개요)
```
[ 검색 입력: "음식명 검색" + 자동완성 드롭다운 ]

[ 기간 선택: Segmented  ( 30일 | 90일 | 전체 )  기본 90일 ]

--- 검색어 없을 때 ---
[ 안내: "음식명을 검색해 보세요" ]
[ 최근 먹은 음식 칩: 신라면 · 닭가슴살 · 바나나 · ... (탭 → 검색) ]

--- 검색 결과 있을 때 ---
[ 빈도 요약 카드 ]
  "이 기간 동안 12번 먹었어요"
  마지막 섭취: 3일 전 (6/20 점심)
  주로 간식으로 7번 · 저녁 3번 · 아침 2번      ← 끼니 분포 한 줄

[ 섭취 이력 리스트 (시간 역순, 행 탭 → 그날 식단으로 이동) ]
  6/20 (목) · 점심 · 신라면 · 480 kcal
  6/18 (화) · 간식(점심·저녁 사이) · 신라면 · 480 kcal
  6/12 (수) · 저녁 · 신라면 · 480 kcal
  ... [더 보기 / 페이지네이션]
```

### 3.3 끼니 표기
- `mealSlot`: 아침 / 점심 / 저녁 / 간식.
- `mealSlot === SNACK`이면 `snackPlacement`(아침 전 / 아침·점심 사이 / 점심·저녁 사이 / 저녁 후)를 보조 표기.
- 레거시 row(`mealSlot === null`)는 끼니 라벨을 생략하거나 "끼니 미지정"으로 표기.

## 4) 결정 사항 (전부 확정)

### 4.1 사용자 1차 결정

| ID | 항목 | 결정 |

### 4.2 추천안 채택 (2026-06-23 확정)

| ID | 항목 | 결정 | 비고 |
|---|---|---|---|
| C-5 | 진입 위치 | **통계(Stats) 탭 상단 "음식 검색" 진입 → 별도 검색 화면(스택 라우트 `FoodSearchScreen`)** | "내 식사 분석" 의미와 통계 탭 정합. 탭 수 5개 유지 |

## 5) 수용 기준 (AC, ATDD-lite)

> 행위·계약·상태 중심. AC ID는 acceptance test와 1:1 추적 가능해야 한다.

### AC-01 이름 검색 결과 (API)
- **Given** 로그인 사용자에게 이름에 "라면"을 포함한 식사 N건이 있을 때
- **When** `GET /me/meals?q=라면&from=...&to=...` 호출
- **Then** 응답 `items`는 이름에 "라면"이 포함된 활성 식사만, `consumedAt` 역순으로 반환하고, `total`은 해당 기간 매칭 건수를 반환한다.

### AC-02 대소문자·공백 무시
- 검색어는 trim 후 `mode: 'insensitive'`로 매칭한다. "라면 "(뒤 공백), "RAMEN"/"ramen"이 동일하게 매칭된다.

### AC-03 기간 필터 정합 (빈도)
- `from`/`to`가 주어지면 그 범위 안의 매칭 건수만 `total`로 집계한다. 기간을 바꾸면 빈도 숫자도 그에 맞게 바뀐다.

### AC-04 끼니·간식 위치 표기
- 이력 각 항목은 `mealSlot`을 라벨로 표시한다. `mealSlot === SNACK`이면 `snackPlacement`를 보조 표기한다. `mealSlot === null`이면 끼니 라벨을 강제하지 않는다.

### AC-05 자동완성
- 검색 입력 중 `GET /me/meal-entry-suggestions?q=...` 후보가 노출되고, 후보 선택 시 그 이름으로 검색이 수행된다.

### AC-06 빈 결과 상태
- 매칭 식사가 0건이면 "기록이 없어요" 류 빈 상태를 표시하고, 빈도는 "0번"으로 표시한다(오류로 처리하지 않음).

### AC-07 권한
- 비로그인/만료 토큰이면 기존 인증 실패 흐름(LoginScreen reset 또는 silent)으로 처리하고, 타 사용자 식사는 절대 노출하지 않는다(서버 `userId` 스코프).

### AC-08 기간 프리셋 (C-6)
- 기간 Segmented는 `30일 / 90일 / 전체` 3개, 기본값 90일.
- 프리셋을 바꾸면 그 기간으로 `from`/`to`가 재계산되어 빈도·이력이 갱신된다. "전체"는 기간 필터 없이 조회한다.

### AC-09 끼니별 분포 요약 (C-7)
- 검색 결과가 1건 이상일 때, 빈도 카드 하단에 끼니별 횟수를 많은 순 최대 3개까지 한 줄로 표시한다(예: `주로 간식 7번 · 저녁 3번`).
- 0건인 끼니 슬롯은 표기에서 생략한다. 결과 0건이면 줄 자체를 숨긴다.

### AC-10 빈 검색어 / 이력 탭 이동 (C-8, C-9)
- 검색어가 없으면 안내 문구 + "최근 먹은 음식" 칩을 노출하고, 칩 탭 시 그 이름으로 검색을 실행한다.
- 이력 항목을 탭하면 해당 식사의 날짜(KST)로 `PastMealBrowseScreen`을 연다.

## 6) API / 백엔드 영향

### 6.1 변경: `GET /me/meals`
- 신규 쿼리 파라미터 `q`(선택). 존재 시 `where.name = { contains: q.trim(), mode: 'insensitive' }` 추가.
- 기존 파라미터(`page`, `size`, `from`, `to`, `mealSlot`, `excludeFoodTemplate`)와 조합 가능.
- 응답 스키마 변경 없음(`items` + `total`을 그대로 사용. `total`이 곧 기간 내 빈도).
- 후방 호환: `q` 미전달 시 기존 동작과 동일.

### 6.2 재사용: `GET /me/meal-entry-suggestions`
- 변경 없음. 자동완성에 그대로 사용.

### 6.3 데이터/마이그레이션
- 스키마 변경 0, 마이그레이션 0. `@@index([userId, consumedAt])` 활용. `name contains`는 인덱스 미적용이나 사용자별 식사량 규모에서 성능 문제 가능성 낮음(필요 시 후속 인덱스 검토).

## 7) 상태 처리 (5+상태)

| 상태 | 표현 |
|---|---|
| 기본 | 검색 입력 + (검색어 있을 때) 빈도 카드 + 이력 리스트 |
| 로딩 | 검색/페이지 로딩 스피너. 자동완성은 디바운스 후 조회 |
| 빈 데이터 | "이 기간에 기록이 없어요" + 빈도 0 (AC-06) |
| 오류 | 네트워크/서버 오류 시 재시도 가능한 오류 메시지(`toUserMessage`) |
| 완료/성공 | 결과 리스트 정상 노출 |
| 권한 제한 | 인증 만료 시 기존 인증 실패 흐름 (AC-07) |

## 8) 디자인 게이트 (65-design-gate)

- 본 기능은 **독립 신규 화면(`FoodSearchScreen`)** 이다(C-5 확정). 이중 디자인안(안 A: 로컬 목업 / 안 B: Stitch)을 **병렬 작성 → 동시 제시 → HUMAN 선택**한 뒤에만 구현에 착수한다.
- 디자인 산출물 필수 포함: 화면 구조, 상태 스펙(기본/로딩/빈/오류/권한), 반응형 기준, 다크모드 토큰 정합.
- `67-dual-design-exemption` 면제 대상 아님(신규 화면 + 검색·기간 Segmented·빈도 카드·이력 리스트 등 복합 구성) → **이중안 진행**.

## 9) 비기능 / 의존성

- 추가 외부 의존성 없음.
- 자동완성은 디바운스 + `AbortSignal`로 요청 취소(기존 `fetchMealEntrySuggestions` 패턴).
- 라이트/다크 토큰 자동 정합, 대비 4.5:1.

## 10) DoD (구현 단계에서 충족)

- AC-01~10 대비 acceptance test 통과(해당 범위).
- 검색·기간 프리셋·빈도·끼니 분포·이력·끼니 표기·자동완성·빈 검색어 칩·이력 탭 이동·빈/오류/권한 상태 모두 동작.
- `q` 추가 후 기존 `GET /me/meals` 사용처(목록·달력 dots 등) 회귀 없음.
- tsc/lint 통과, 라이트/다크 점검.

## 11) Gate 점검 메모

- Gate 1: 본 PRD(목적·범위·흐름·정책) + AC(§5) + API 계약 초안(§6) 확보. 결정 사항 전부 확정(C-1~C-9). 화면 기준은 디자인 단계 산출.
- Gate 2(병렬): API 계약(`q` 추가) 고정 + 상태 UI 정의 + 디자인 승인 후 `frontend-agent`(모바일 `FoodSearchScreen` + 통계 탭 진입점) / `backend-agent`(`/me/meals` q 필터) 병렬 가능. 파일 충돌 낮음(서버 `routes/me.ts` 단일 지점 vs 모바일 신규 화면).
